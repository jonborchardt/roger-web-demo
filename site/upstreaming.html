Bringing Roger to ScummVM

Why

Roger runs original commercial games from their original data and enhances only how they are displayed. It is opt-in and byte-identical to stock ScummVM when disabled. It is not a new engine, a reimplementation, or a private game — it is a display-layer provider inside the existing SCI engine. Game detection stays SCI's; no new detection tables, no metaengine, no duplicated engine.

That shape makes it a low-risk addition: players who want the enhancement turn it on; everyone else is unaffected.

How

All SCI graphics chokepoints call a single neutral, engine-owned observer interface. Roger lives entirely behind that seam — the fork's changes outside engines/sci/roger/ contain zero Roger references and are byte-identical to stock when the observer is null. This keeps the maintained diff small and the change reviewable.

The plan is to talk first (ScummVM has a strong discuss-before-PR culture), then submit one coherent engine-seam change as a few clean commits, plus small separable PRs for the generic pieces (a headless scripted-input driver, a page-relative data-fetch portability fix). Clean PR branches are built fresh from upstream master; the fork itself remains a long-lived showcase either way.

Try it first

The live browser demo is the fastest way to see exactly what "display-only enhancement" means — press F10 to flip between Enhanced, Original, and Side-by-Side.
